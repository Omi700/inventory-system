# 商品进销存管理系统需求分析（两周 MVP 版）

## 1. 项目定位

本项目是一个两周内完成的商品进销存管理系统。系统不追求企业级 ERP 的完整复杂度，而是优先完成一个可运行、可演示、技术亮点明确的 Web 项目。

基础功能保留进销存核心闭环：用户登录、权限控制、商品管理、仓库管理、入库、出库、库存查询和简单报表。特色功能保留三个重点：

1. 接入大模型的自然语言业务查询 Text-to-SQL。
2. 基于 Redis + Lua 的高并发防超卖库存扣减。
3. 基于 WebSocket 的多仓实时库存大屏。

为了保证两周内能完成，系统采用轻量化设计：

- 权限模块只做两个角色，不做复杂权限点配置。
- 商品图片支持多图上传，但只做本地存储、主图展示和详情图库，不做裁剪、对象存储。
- Text-to-SQL 接入大模型，但只允许查询固定业务表，并必须经过 SQL 安全校验。
- Redis 防超卖先做成可演示的并发测试模块，不接完整订单系统。
- WebSocket 大屏只展示核心库存指标、图表和实时事件。
- 后端采用 Java + Python 双服务：Spring Boot 负责主业务系统，Python 负责大模型 Text-to-SQL 能力。

## 2. 项目目标

### 2.1 总体目标

- 完成一个可运行的进销存 Web 系统。
- 使用 MySQL 保存核心业务数据。
- 使用 Redis 和 Lua 脚本实现并发安全的库存扣减演示。
- 使用 WebSocket 实现库存变化实时推送。
- 接入大模型，实现受限场景下的自然语言转 SQL 查询。
- 提供简洁但完整的页面，能支撑项目汇报演示。

### 2.2 两周完成标准

- 系统支持两个角色：系统管理员、仓库管理员。
- 系统管理员可以管理用户、商品、仓库和查看全部功能。
- 仓库管理员可以操作入库、出库、库存查询、报表和实时大屏。
- 商品支持上传并展示多张图片。
- 入库、出库后库存能正确变化，并写入库存流水。
- Text-to-SQL 可以通过 Python AI 服务调用大模型生成查询 SQL，并由 Java 主服务返回结果和中文总结。
- Redis 防超卖模块能证明并发扣减时库存不会小于 0。
- WebSocket 大屏能在库存变化后实时刷新。

## 3. 用户角色与权限

### 3.1 角色设计

系统设置两个角色。

#### 系统管理员

系统管理员负责基础资料和系统配置。

权限范围：

- 登录系统。
- 管理用户账号。
- 管理商品资料，包括商品图片。
- 管理仓库资料。
- 查看库存、入库、出库和报表。
- 使用 Text-to-SQL 查询。
- 查看实时大屏。
- 执行 Redis 防超卖测试。

#### 仓库管理员

仓库管理员负责日常库存业务操作。

权限范围：

- 登录系统。
- 查看商品资料。
- 查看仓库资料。
- 创建入库记录。
- 创建出库记录。
- 查询库存。
- 查看简单报表。
- 使用 Text-to-SQL 查询。
- 查看实时大屏。

仓库管理员不能：

- 新增、修改、禁用用户。
- 新增、修改、停用仓库。
- 执行 Redis 防超卖测试。
- 修改系统配置。

### 3.2 权限实现边界

两周版本做“角色级权限”，不做细粒度权限点。

实现方式：

- 用户表保存角色字段。
- 登录后后端签发 token，并在 token 中保存用户角色。
- 前端根据角色显示不同菜单。
- 后端接口根据角色做权限校验。
- 权限规则写在后端代码配置中，不单独做权限配置页面。

### 3.3 用户管理

仅系统管理员可以管理用户。

功能要求：

- 新增用户。
- 修改用户昵称、密码和角色。
- 禁用用户。
- 查询用户列表。

缩减说明：

- 暂不做部门。
- 暂不做用户头像。

## 5. 特色功能需求

### 5.1 特色功能一：接入大模型的 Text-to-SQL

#### 5.1.1 功能描述

用户可以在系统中输入中文业务问题。Java 主服务接收问题并调用 Python AI 服务，Python AI 服务调用大模型理解问题并生成 SQL。Java 主服务对 SQL 做安全校验，校验通过后查询 MySQL，并返回查询结果和中文总结。

示例问题：

- 上个月销量最好的商品有哪些？
- 最近 7 天出库最多的是哪几款商品？
- 目前北京仓库还有多少库存？
- 哪些商品低于安全库存？
- 各仓库当前库存总量是多少？

#### 5.1.2 大模型接入方式

第一版采用 Java + Python 双服务接入大模型。

服务分工：

- Java Spring Boot 主服务：负责登录权限、业务接口、SQL 安全校验、MySQL 查询执行、查询日志保存。
- Python AI 服务：负责调用大模型，根据表结构和用户问题生成 SQL，并辅助生成中文总结。
- 前端只调用 Java 主服务，不直接调用 Python AI 服务。

Python AI 服务通过环境变量配置大模型参数。

建议配置项：

- `LLM_API_KEY`
- `LLM_BASE_URL`
- `LLM_MODEL`

系统向大模型提供以下上下文：

- 可查询的数据表说明。
- 字段含义。
- 表之间关系。
- 当前用户问题。
- SQL 生成规则。

大模型返回结构化 JSON，避免自由文本难以解析。

示例返回：

```json
{
  "sql": "SELECT p.product_name, SUM(s.quantity) AS total_out FROM stock_out s JOIN product p ON s.product_id = p.id WHERE s.created_at >= DATE_SUB(CURDATE(), INTERVAL 7 DAY) GROUP BY p.id, p.product_name ORDER BY total_out DESC LIMIT 10",
  "answer_style": "ranking",
  "reason": "用户想查询最近 7 天出库最多的商品"
}
```

#### 5.1.3 SQL 安全要求

Text-to-SQL 是本项目风险最高的功能，必须做限制。

安全要求：

- 只允许执行 `SELECT`。
- 禁止 `INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE`。
- 禁止多语句执行。
- 禁止查询非白名单表。
- 必须自动添加或检查 `LIMIT`。
- 默认最多返回 20 条。
- 执行 SQL 前做语法和关键词校验。
- 保存用户问题、模型返回 SQL、安全校验结果、执行结果摘要。
- Python AI 服务不直接连接 MySQL，不直接执行 SQL。
- SQL 执行权只保留在 Java 主服务中。

#### 5.1.4 第一版查询范围

为了保证两周内可控，第一版只允许查询以下业务范围：

- 商品库存。
- 仓库库存。
- 入库记录。
- 出库记录。
- 库存流水。
- 低库存商品。
- 最近 7 天或 30 天的出库排行。

不允许查询：

- 用户密码。
- token。
- 系统配置。
- 非业务白名单表。

#### 5.1.5 页面展示

Text-to-SQL 页面需要展示：

- 用户输入的问题。
- 大模型生成的 SQL。
- 安全校验状态。
- 查询结果表格。
- 中文总结。

### 5.2 特色功能二：基于 Redis 的高并发防超卖库存扣减

#### 5.2.1 功能描述

该功能用于模拟热门商品在高并发出库或抢单场景下的库存扣减问题。系统将某个商品的库存加载到 Redis 中，通过 Lua 脚本完成原子扣减，保证并发请求下库存不会变成负数。

#### 5.2.2 两周版实现边界

第一版做成一个独立的“防超卖测试模块”，不接完整订单系统。

功能要求：

- 只有系统管理员可以执行防超卖测试。
- 系统管理员选择一个商品和仓库。
- 设置 Redis 初始库存。
- 设置并发请求数量和每次扣减数量。
- 系统并发请求 Redis 扣减库存。
- Redis Lua 脚本保证库存充足才扣减。
- 页面展示测试结果：总请求数、成功数、失败数、Redis 剩余库存、是否发生超卖。
- 扣减成功的记录写入 MySQL 扣减流水表。

#### 5.2.3 Lua 扣减逻辑

伪代码：

```text
读取 Redis 库存
如果库存不存在，返回失败
如果库存小于扣减数量，返回库存不足
否则扣减库存
记录扣减成功
返回成功
```

#### 5.2.4 验收重点

- 并发请求数大于库存数量时，成功扣减数不能超过库存。
- Redis 库存不能小于 0。
- 页面能清晰展示并发测试结果。
- 能说明 Redis + Lua 为什么能避免超卖。

### 5.3 特色功能三：基于 WebSocket 的多仓实时库存大屏

#### 5.3.1 功能描述

系统提供一个实时库存大屏。当发生入库、出库、库存预警或 Redis 扣减事件时，后端通过 WebSocket 将消息主动推送到前端，大屏实时更新库存数据和事件列表。

#### 5.3.2 展示内容

两周版本的大屏保留关键展示项：

- 当前商品总数。
- 当前仓库总数。
- 当前库存总量。
- 低库存预警数量。
- 各仓库库存柱状图。
- 最近库存变动事件列表。
- 低库存预警列表。
- 热门商品缩略图和库存变化。

#### 5.3.3 推送事件

后端在以下场景推送消息：

- 入库成功。
- 出库成功。
- Redis 扣减测试成功或失败。
- 商品库存低于安全库存。

#### 5.3.4 消息格式示例

```json
{
  "eventType": "STOCK_OUT",
  "warehouseName": "北京仓",
  "productName": "埃塞俄比亚咖啡豆",
  "productImage": "/uploads/products/coffee-bean.png",
  "changeQuantity": 5,
  "currentStock": 18,
  "message": "北京仓的埃塞俄比亚咖啡豆出库 5 件，当前库存 18 件"
}
```

#### 5.3.5 验收重点

- 前端能连接 WebSocket。
- 入库或出库后，大屏数据能自动刷新。
- 低库存时能出现实时预警。
- 页面至少包含 1 个 ECharts 图表和 1 个实时事件列表。

## 7. 非功能需求

### 7.1 性能需求

- 普通列表查询响应时间控制在 1 秒以内。
- 入库、出库操作响应时间控制在 2 秒以内。
- WebSocket 推送延迟尽量控制在 1 秒以内。
- Redis 防超卖测试至少支持 100 个并发请求。
- Text-to-SQL 查询响应时间受大模型影响，第一版控制在 15 秒以内即可。

### 7.2 安全需求

- 管理页面必须登录后访问。
- 后端接口必须检查角色权限。
- 商品图片上传必须限制文件类型、单张大小和单商品图片数量。
- Text-to-SQL 只能执行查询类 SQL。
- 大模型生成 SQL 后必须经过安全校验。
- Redis 扣减测试仅系统管理员可用。
- 关键库存操作必须记录流水。

### 7.3 可靠性需求

- MySQL 是最终库存数据来源。
- Redis 只用于高并发扣减演示和临时库存。
- 入库、出库必须保证库存和流水同时更新。
- WebSocket 断开后前端应能自动重连。
- 大模型调用失败时，页面应提示失败原因，不影响其他功能使用。

## 9. 技术方案建议

### 9.1 后端

后端采用 Java + Python 双服务。

Java 主服务：

- Spring Boot。
- Spring Security 或 JWT 自定义认证。
- MyBatis-Plus。
- MySQL。
- Redis。
- WebSocket。

Python AI 服务：

- FastAPI。
- 大模型 API 客户端。
- Prompt 构建。
- SQL JSON 生成。
- 查询结果中文总结。

分工理由：

- Spring Boot 更适合承载权限、事务、库存扣减、Redis、WebSocket 等企业业务逻辑。
- Python 更适合快速接入大模型、调试 Prompt 和扩展智能体能力。
- 前端只访问 Java 主服务，系统边界更清晰。

### 9.2 前端

建议使用：

- Vue3。
- Element Plus。
- ECharts。
- WebSocket 客户端。

### 9.3 数据库与中间件

建议使用：

- MySQL：保存用户、商品、仓库、库存、入库、出库、流水等数据。
- Redis：保存高并发扣减测试中的临时库存。
- Lua：保证 Redis 扣减操作原子执行。

### 9.4 大模型 Text-to-SQL

建议采用“Java 主服务接收问题 + Python AI 服务生成 SQL + Java 主服务安全校验 + Java 主服务执行 SQL + Python 或 Java 生成总结”的方式。

核心步骤：

- 前端提交自然语言问题。
- Java 主服务检查用户权限。
- Java 主服务将问题和受限表结构说明发送给 Python AI 服务。
- Python AI 服务调用大模型生成 SQL。
- Python AI 服务返回结构化 JSON。
- Java 主服务校验 SQL 是否安全。
- Java 主服务执行 SQL。
- Java 主服务将结果发送给 Python AI 服务生成中文总结，或使用 Java 规则生成总结。
- 前端展示 SQL、表格结果和总结。

## 10. 典型业务流程

### 10.1 登录与权限流程

1. 用户输入用户名和密码。
2. 系统校验账号状态和密码。
3. 登录成功后返回 token 和角色。
4. 前端根据角色展示菜单。
5. 后端接口根据角色判断是否允许访问。

### 10.2 商品图片上传流程

1. 系统管理员新增或编辑商品。
2. 上传一张或多张商品图片。
3. 后端校验图片格式、单张大小和数量。
4. 图片保存到本地上传目录。
5. 数据库保存图片访问路径、主图标记和排序。
6. 如果未手动设置主图，系统默认将第一张图片设为主图。
7. 商品列表和库存页面展示商品主图。
8. 商品详情页展示全部图片。

### 10.3 入库流程

1. 用户选择商品、仓库和入库数量。
2. 系统新增入库记录。
3. 系统增加当前库存。
4. 系统写入库存流水。
5. 系统通过 WebSocket 推送库存变化。
6. 大屏实时刷新。

### 10.4 出库流程

1. 用户选择商品、仓库和出库数量。
2. 系统检查当前库存是否充足。
3. 库存充足则新增出库记录。
4. 系统扣减当前库存。
5. 系统写入库存流水。
6. 如果库存低于安全库存，生成预警消息。
7. 系统通过 WebSocket 推送库存变化和预警。

### 10.5 Text-to-SQL 查询流程

1. 用户输入自然语言问题。
2. Java 主服务校验登录状态和功能权限。
3. Java 主服务读取可查询表结构说明。
4. Java 主服务调用 Python AI 服务。
5. Python AI 服务调用大模型生成 SQL。
6. Java 主服务校验 SQL 安全性。
7. 校验通过后由 Java 主服务执行 SQL。
8. Java 主服务调用 Python AI 服务生成中文总结，或使用规则总结。
9. 系统展示生成 SQL、查询结果和中文总结。
10. Java 主服务保存查询日志。

### 10.6 Redis 防超卖测试流程

1. 系统管理员选择商品和仓库。
2. 系统将测试库存写入 Redis。
3. 系统管理员设置并发请求数量和每次扣减数量。
4. 系统并发调用 Redis Lua 脚本扣减库存。
5. 系统统计成功和失败请求。
6. 系统展示最终库存和是否超卖。

### 10.7 WebSocket 大屏流程

1. 用户打开实时大屏。
2. 前端与后端建立 WebSocket 连接。
3. 入库、出库或 Redis 扣减发生后，后端推送事件。
4. 前端实时刷新统计卡片、图表和事件列表。

## 12. 验收标准

### 12.1 权限验收

- 系统至少有系统管理员和仓库管理员两个角色。
- 系统管理员可以管理用户。
- 仓库管理员不能进入用户管理页面。
- 仓库管理员不能执行 Redis 防超卖测试。
- 无权限访问后端接口时，系统返回权限不足。

### 12.2 基础功能验收

- 可以新增、修改、查询商品。
- 商品可以上传并展示多张图片。
- 商品列表能展示主图，商品详情能展示图片图库。
- 可以新增、修改、查询仓库。
- 可以完成入库和出库。
- 当前库存能正确变化。
- 库存流水能正确记录。

### 12.3 Text-to-SQL 验收

- 系统能调用大模型生成 SQL。
- 至少支持 5 类自然语言业务问题。
- 页面能展示模型生成的 SQL。
- 页面能展示查询结果。
- 系统能用中文总结结果。
- 修改类 SQL 会被拦截，不会执行。

### 12.4 Redis 防超卖验收

- 可以设置并发请求数量。
- 可以模拟多个请求同时扣减同一商品。
- 最终库存不会小于 0。
- 成功扣减总数不会超过初始库存。
- 页面能展示成功数、失败数和剩余库存。

### 12.5 WebSocket 大屏验收

- 前端能建立 WebSocket 连接。
- 入库后大屏能实时变化。
- 出库后大屏能实时变化。
- 低库存时大屏能显示预警。
- 大屏至少包含一个图表和一个实时事件列表。

## 13. 项目亮点

本项目在控制工作量的前提下，保留了四个能在汇报中讲清楚的亮点：

- 两角色权限模块：让系统具备基础管理系统形态。
- 商品多图管理：让商品资料更完整，页面展示不只是表格数据。
- 大模型 Text-to-SQL：体现自然语言交互和数据库查询结合。
- Redis 防超卖：体现 NoSQL、高并发和 Lua 原子操作。
- WebSocket 大屏：体现网络编程、实时通信和数据可视化。

基础功能负责提供真实业务数据，特色功能负责展示技术深度。这样既能保证两周内完成，又不会做成普通的进销存 CRUD 项目。
